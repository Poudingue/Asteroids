# Fixed-dt Deterministic Simulation Mode ‚Äî Design Spec

**Date**: 2026-03-30
**Status**: Approved
**Scope**: Time stepping, deterministic RNG, scenario system, CLI interface

---

## Overview

Add a fixed-timestep deterministic simulation mode to the Asteroids engine. This enables reproducible physics, scripted test scenarios, recorded session replay, state snapshot comparison, and headless simulation for automated testing and future ML pipelines.

---

## Time Stepping

### `SimulationMode` enum (added to `TimeConfig`)

- **`RealTime`** ‚Äî Current behavior. dt from wall clock (`Instant::elapsed`), capped at `MAX_DT`. Default mode.
- **`FixedInteractive(target_fps: u32)`** ‚Äî Fixed dt = `1.0 / target_fps`. If a frame computes faster than dt, sleep the difference. Slowdowns happen naturally. Playable, deterministic.
- **`FixedFullSpeed(target_fps: u32)`** ‚Äî Fixed dt = `1.0 / target_fps`. No sleeping, no waiting. Frames advance as fast as CPU allows. For replay and testing. Window visible but likely unplayable at high speed.
- **`Headless(target_fps: u32)`** ‚Äî Fixed dt, no window, no renderer, no GPU. Pure simulation loop. Maximum speed.

In all fixed modes, time advances by `globals.time.time_current_frame += 1.0 / target_fps` instead of reading from `Instant::elapsed`.

Headless mode skips SDL2 video subsystem init, window creation, wgpu instance/device/surface, and the entire render path. The game loop calls `update_game` in a tight loop until the scenario's `run_until` frame.

## Deterministic RNG

Currently `GameState` uses `ThreadRng` ‚Äî not seedable, not reproducible.

**Change:** Replace `ThreadRng` with `SmallRng` everywhere. `SmallRng` is fast, seedable, and `rand` recommends it for games. Both implement `Rng`, so all call sites are unchanged.

- **`RealTime` mode**: `SmallRng::from_entropy()` ‚Äî random seed, non-reproducible (same behavior as before).
- **Fixed-dt modes**: `SmallRng::seed_from_u64(seed)` ‚Äî seed comes from the scenario file or CLI argument.

This is a type change on the `rng` field in `GameState` plus updating the constructor. No other code changes needed.

## Scenario File Format

### Two file types per scenario

**`.ron` ‚Äî Scenario definition** (sparse, human-editable):

```ron
Scenario(
    name: "chain_reaction_test",
    seed: 42,
    target_fps: 60,
    mode: FullSpeed,  // FullSpeed | Interactive | Headless
    setup: [
        SpawnAsteroid(pos: (500.0, 300.0), radius: 80.0, velocity: (10.0, -5.0)),
        SpawnAsteroid(pos: (600.0, 350.0), radius: 60.0, velocity: (0.0, 0.0)),
        SetShipPosition(400.0, 300.0),
    ],
    actions: [
        (frame: 30, AimAt(0.52)),
        (frame: 60, Fire),
        (frame: 90, MoveDirection(1.0, 0.0)),
        (frame: 120, Teleport),
        (frame: 120, StopMoving),
    ],
    run_until: 300,
    snapshots_at: [60, 120, 180, 300],
    trajectory_interval: 10,
    assertions: [
        AtFrame(300, ObjectCountAtLeast(4)),
        AtFrame(300, ShipAlive),
    ],
)
```

**`.inputs` ‚Äî Dense input recording** (binary, zstd-compressed):

- Header: seed (u64), target_fps (u32), frame_count (u64)
- Per-frame record: 4√óf32 (left_stick_x, left_stick_y, right_stick_x, right_stick_y) + u16 button bitfield
- Compressed with zstd ‚Äî high temporal correlation between adjacent frames yields excellent compression ratios
- Generated by recording a real play session, or synthesized from `.ron` actions
- A scenario's `.ron` file can reference an `.inputs` file via an `input_file` field, used instead of (or in addition to) the `actions` list

### Output files

- `.snapshot.N` ‚Äî Serialized `GameState` at frame N (RON format for human inspection, bincode for binary comparison)
- `.trajectory` ‚Äî Per-entity log: `(frame, entity_id, x, y, vx, vy, health, radius)` ‚Äî CSV or binary

## Rust Test API

### Loading and running scenarios

```rust
use asteroids::scenario::{Scenario, ScenarioResult};

#[test]
fn test_chain_reaction() {
    let result = Scenario::load("scenarios/chain_reaction.ron").run();
    assert!(result.final_state.objects.len() >= 4);
    assert!(result.final_state.ship.health > 0.0);
}

#[test]
fn test_determinism() {
    let result_a = Scenario::load("scenarios/chain_reaction.ron").run();
    let result_b = Scenario::load("scenarios/chain_reaction.ron").run();
    assert_eq!(result_a.snapshots, result_b.snapshots); // byte-identical
}

#[test]
fn test_replay_matches_recording() {
    let recorded = Scenario::load("scenarios/recorded_session.ron").run();
    let replayed = Scenario::load("scenarios/recorded_session.ron").run();
    let diff = recorded.diff(&replayed);
    assert!(diff.is_empty(), "Divergence at: {:?}", diff);
}
```

### `ScenarioResult` contains

- `final_state: GameState` ‚Äî state at `run_until` frame
- `snapshots: HashMap<u64, StateSnapshot>` ‚Äî full state at requested frames
- `trajectories: Vec<TrajectoryEntry>` ‚Äî logged entity data

### `StateSnapshot`

Implements `PartialEq` and `Serialize`. Two snapshots are equal if and only if the simulation is deterministic. The `diff()` method returns the first frame and field where they diverge.

### Programmatic scenarios (builder API, no file)

```rust
#[test]
fn test_explosion_push() {
    let result = Scenario::builder()
        .seed(42)
        .fps(60)
        .spawn_asteroid((500.0, 300.0), 80.0)
        .at_frame(10, Action::Fire)
        .run_until(60);
    // assert on result...
}
```

## CLI Interface

```bash
# Normal game (unchanged default)
asteroids

# Run a scenario interactively (playable, fixed dt)
asteroids --scenario scenarios/chain_reaction.ron

# Run full-speed (window visible, as fast as CPU allows)
asteroids --scenario scenarios/chain_reaction.ron --full-speed

# Run headless (no window, no GPU, max speed)
asteroids --scenario scenarios/chain_reaction.ron --headless

# Record a play session
asteroids --record session.inputs --seed 42 --fps 60

# Override mode from CLI (takes precedence over .ron file)
asteroids --scenario scenarios/test.ron --headless --fps 120
```

**Implementation:** `clap` crate for argument parsing ‚Äî derives from struct, standard in the Rust ecosystem.

### Headless mode in `main.rs`

- Skip `sdl_context.video()`, window creation, wgpu init, renderer
- Keep SDL2 base init only if controller input is needed (not for headless scenarios)
- Game loop: `loop { apply_inputs(frame); update_game(&mut state, &mut globals); frame += 1; if frame >= run_until { break; } }`
- Write snapshots/trajectories to disk, exit

## Module Structure

### New files

| File | Responsibility |
|------|----------------|
| `src/scenario.rs` | `Scenario`, `ScenarioResult`, `StateSnapshot`, `Action` enums, builder API, load/run logic |
| `src/time.rs` | `SimulationMode` enum, `FixedTimeSource` ‚Äî extracted from `TimeConfig` in `parameters.rs` |
| `src/recording.rs` | Dense input recording/playback ‚Äî read/write `.inputs` files, zstd compress/decompress |

### Modified files

| File | Change |
|------|--------|
| `src/main.rs` | CLI parsing (clap), branch on mode (normal / scenario / headless), recording hooks |
| `src/game.rs` | `SmallRng` instead of `ThreadRng`, `GameState` derives `Serialize`/`Clone` for snapshots |
| `src/parameters.rs` | `SimulationMode` added to `TimeConfig` |
| `src/lib.rs` | Export new modules |
| `Cargo.toml` | Add deps: `clap`, `ron`, `serde` (derive), `zstd`, update `rand` usage for `SmallRng` |

### Not modified

Rendering, physics, input modules ‚Äî the scenario system wraps around the existing game loop, it doesn't reach into it.

## Dependencies on Other Systems

- **Spatial field effects** (planned): scenarios can test field behavior deterministically ‚Äî spawn explosions, verify wind propagation frame-by-frame
- **Visual regression**: headless mode + configurable resolution enables offline rendering for pixel-diff comparison, supersampled rendering, ML training data generation
- **Input recording/replay**: the `.inputs` format and `recording.rs` module provide the foundation for in-game replay features
